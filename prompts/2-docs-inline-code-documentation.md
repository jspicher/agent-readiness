## Original Signal Evaluation Criteria

The agent readiness report evaluated this signal using these instructions:

Inline code documentation -- the repo's public surface area carries machine-readable doc comments in the language's canonical format, AND a coverage gate prevents the surface from rotting. PASS requires BOTH of the following:

1. **Doc comments exist on the public API.** Sample 10-20 exported symbols across the repo and confirm each carries a real doc comment in the canonical format for the language:
   - **TypeScript / JavaScript**: TSDoc / JSDoc `/** ... */` blocks immediately above exported `function`, `class`, `interface`, `type`, `const`. Tags: `@param`, `@returns`, `@throws`, `@example`, `@deprecated`. (Reference: https://tsdoc.org/ and https://jsdoc.app/)
   - **Python**: PEP 257 docstrings (triple-quoted string as first statement) on every public module, class, function, and method. Style consistent across the repo (Google, NumPy, or reST -- pick one and stick to it).
   - **Go**: GoDoc comments -- full sentences starting with the symbol name, immediately above every exported identifier (`func Foo`, `type Bar`, `var Baz`). Package comment in `doc.go` or above the `package` clause of one file.
   - **Rust**: `///` outer doc comments on public items (`pub fn`, `pub struct`, `pub trait`); `//!` inner doc comments at the top of each `pub mod`. Examples in `/// # Examples` blocks compile via `cargo test --doc`.
   - **Java / Kotlin**: Javadoc / KDoc `/** ... */` on every `public` class and method with `@param`, `@return`, `@throws`.
   - **C# / .NET**: XML doc comments (`/// <summary>`) on every public type and member; project compiled with `<GenerateDocumentationFile>true</GenerateDocumentationFile>`.

2. **A coverage gate is wired into CI** that fails the build when documentation drops below a project-tuned threshold. A linter that's installed but never invoked is not a gate.

A FAIL looks like: zero JSDoc blocks in `src/**`; Python modules where most functions lack docstrings; Go packages with no exported-symbol comments; doc comments present in one corner of the repo but absent everywhere else; a configured linter that nobody runs; a coverage tool installed in `package.json` but no CI step calling it; docstrings that simply restate the function name (`def get_user(id): """Get a user by id."""`); autogenerated stub blocks left with `TODO` bodies.

## CRITICAL: Quality Standards

Your fix must **genuinely improve the codebase**. Do NOT use workarounds or shortcuts:

- **NO** doc comments that restate the function name. `/** Gets the user. */ function getUser()` is noise -- the agent already reads the signature. Write the *why*, the *constraints*, the *side effects*.
- **NO** AI-generated stub blocks with empty bodies (`/** @param id TODO */`, `"""TODO: document this."""`). Empty placeholders score worse than no comment because they pass the coverage gate while teaching the agent nothing.
- **NO** documenting only private/internal symbols while leaving the public API bare. The public surface is what callers (and agents) consume.
- **NO** coverage tool installed without a CI step that invokes it with `--fail-under` (or equivalent). A linter that never runs is dead code.
- **NO** setting `--fail-under=10` to make the gate trivially pass. Pick a threshold that reflects the current real coverage (run the tool first to measure) and ratchet up over time.
- **NO** mass-suppressing the linter with `// eslint-disable-next-line jsdoc/require-jsdoc` or `# noqa: D` on every file. If the rule is too strict, narrow its `files` glob; do not blanket-disable.
- **NO** stripping doc comments at build time without preserving them somewhere the agent can read (e.g. production source maps that drop comments + no published `.d.ts`). If you minify, ship a typings/doc artifact alongside.
- **NO** introducing a second docstring style next to an existing one. Pick one convention per language and migrate, don't fork.

Examples of BAD fixes:
- `/** Returns the result. */ function compute(a, b) { ... }` -- restates the name, says nothing about what `a` and `b` mean or what "result" is.
- `"""TODO: add docstring."""` committed to satisfy `interrogate`.
- Adding `eslint-plugin-jsdoc` to `devDependencies`, but no `eslint .` step in CI -- the gate doesn't exist.
- Setting `interrogate --fail-under=5` so the bar is below current coverage. The number must reflect reality, not theatre.
- Documenting `src/utils/internal/*.ts` while `src/api/index.ts` (the actual public entry) has no comments.
- Converting half the repo's reST docstrings to Google style and leaving the other half. Now both tooling and humans have to handle two conventions.
